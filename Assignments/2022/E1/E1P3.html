2. E1P3 Statement of Work

2.1 Primary Objectives

The primary objective of this assessment is to determine how well you can layer a component by using a provided internal representation and correspondence contract.

2.2 Specifics

Steps:

  1. Start Visual Studio
  2. Open project E1P3D
  3. Find and open List6.hpp
  4. Required: Implement the operations from List6 that correspond to the 24 test cases shown to the right. Your goal is to have all 24 test cases passs
  5. Commit and Push your changes, which includes List6.hpp

Important Restrictions:

  1. Obey the correspondence - When implementing the member functions you must obey the correspondence internal contract
    That means:
    • assume that the correspondence holds when the member function is called
    • guarantee that the correspondence holds when the member function exits
  2. Use Design by Contract - No defensive programming of List's operations
    • What this means is that the implementation of a List operation must not contain code that checks to make sure that the List operation was called correctly by the client
  3. No Cross Calling - None of List's member functions are permitted to make calls to any of the other public List member functions, i.e., no cross calling
    • For example, under this rule it is illegal for the moveToFinish operation to call any other List operation, e.g., advance
    • All List operations should directly manipulate List's data members
  4. Calling Down - is encouraged (but not required) by creating private member functions that capture code that starts to appear in multiple different places in your List6 implementation. Do this to avoid the duplication of code
  5. No Recursion - None of List's member functions are permitted to be recursive
  6. About Making Copies -
    • When implementing List6's operator = you must use PreemptableQueue's operator = to make a copy of the data members. Otherwise do not call operator = with the List6 data members or with variables of type T.

3. Testing

  • E1P3D testing
    • E1P3D's test cases test against the List's internal representation, so these unit tests will only work with this particular implementation
    • Look at these unit tests to better understand the correspondence between a List's abstract value and its internal representation
    • A number of the provided test cases are minimal, i.e., do not test edge cases (boundary conditions), or many steady-state cases, so you might want to augment this set of test cases with some of your own
